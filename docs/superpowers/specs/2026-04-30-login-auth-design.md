# Login / Logout for Etherscope

**Status:** design approved (2026-04-30)
**Scope:** v1 single-account session auth covering both API and dashboard UI.

## Goal

Replace the current static `API_AUTH_TOKEN` bearer guard with a real
username + password login flow. After this change, both the dashboard UI and
every protected API endpoint require an authenticated session; `/api/health`
remains public.

## Non-goals (v1)

- Multi-user accounts, sign-up, email verification, password reset, 2FA, OAuth.
- Per-route role / permission checks.
- Browser-facing WebSocket authentication (no such endpoint exists today).

## User model

One account, configured via env:

- `AUTH_USERNAME` — e.g. `admin`.
- `AUTH_PASSWORD_HASH` — argon2id hash, generated by a CLI helper.

The plaintext password never sits in `.env`.

## Session model

- Opaque session IDs: 32 random bytes, base64url-encoded.
- Stored server-side in Redis under `session:<id>` with value = username.
- TTL: **24 hours, fixed** (no sliding expiry, no remember-me).
- Cookie: `etherscope_session`, `HttpOnly`, `SameSite=Lax`, `Path=/`,
  `Secure` in production (controlled by `SESSION_COOKIE_SECURE`, default `true`).

## CSRF

The frontend is served same-origin with the API behind the reverse proxy. With
`SameSite=Lax`, browsers will not send the cookie on cross-site `POST`s from
attacker-controlled origins. v1 therefore does not ship a separate CSRF token.
If a future change introduces cross-origin browser clients, a double-submit
CSRF token will be added.

## Brute-force protection

Per-IP login rate limit, Redis-backed:

- Key: `login_fail:<ip>`.
- On failed login: `INCR`, set 15-minute expiry on first hit.
- If counter > 10: respond `429 Too Many Requests` with `Retry-After: 900`.
- Successful logins do not reset the counter (it just expires naturally).

## API surface

All endpoints live under `/api/auth`. The router itself is mounted without
`AuthDep`; per-endpoint requirements are called out below.

### `POST /api/auth/login`

Request body (JSON): `{"username": str, "password": str}`.

Behaviour:
1. If the IP is rate-limited → 429.
2. Compare `username` to `AUTH_USERNAME` and verify password against
   `AUTH_PASSWORD_HASH` using argon2.
3. On mismatch (either field) → increment the per-IP fail counter and return
   `401 {"detail": "invalid credentials"}`. The same generic message is used
   for unknown username and wrong password (no enumeration).
4. On success → create a session in Redis, set the cookie on the response,
   return `200 {"username": "<name>"}`.

### `POST /api/auth/logout`

Reads the cookie, deletes the matching Redis entry, clears the cookie via
`Set-Cookie: etherscope_session=; Max-Age=0`. Returns `204` regardless of
whether a valid session was present (idempotent; no information leak).

### `GET /api/auth/me`

Requires session. Returns `200 {"username": "<name>"}` or `401` if the cookie
is missing / unknown / expired. Used by the frontend to decide between the
login page and the dashboard.

### Existing routers

- `/api/health` stays public.
- All other routers (`price`, `flows`, `whales`, `alerts`, `network`,
  `derivatives`, `leaderboard`) continue to mount with `dependencies=[AuthDep]`,
  but `AuthDep` is rebuilt to mean "valid session cookie."

## Backend implementation outline

### `backend/app/core/sessions.py` (new)

Thin wrapper around the existing Redis client:

- `create_session(username: str) -> str`
- `get_session_username(session_id: str) -> str | None`
- `destroy_session(session_id: str) -> None`

TTL constant `SESSION_TTL_SECONDS = 86400`.

### `backend/app/core/auth.py` (rewrite)

- `hash_password(plain: str) -> str` — argon2id (`argon2-cffi`).
- `verify_password(plain: str, hashed: str) -> bool`.
- `require_auth(request: Request) -> str` — reads
  `request.cookies["etherscope_session"]`, calls `get_session_username`, raises
  `401` on miss. Returns the username so handlers can reference it if needed.
- `RateLimiter` helper: `check_login_ip(ip: str) -> None` (raises `429`),
  `register_login_failure(ip: str) -> None`.
- `AuthDep = Depends(require_auth)` retained as the public symbol.
- The `api_auth_token` legacy bearer path is **deleted**.

### `backend/app/api/auth.py` (new router)

Endpoints described above. Cookie writes go through a shared helper
`_set_session_cookie(response, session_id)` so flags stay consistent.

### `backend/app/core/config.py`

Settings additions:

```python
auth_username: str | None = None
auth_password_hash: str | None = None
session_cookie_secure: bool = True
```

Remove `api_auth_token`.

### `backend/app/main.py`

- Mount the new auth router (public).
- Keep `health` public.
- Keep all other routers with `dependencies=[AuthDep]` (semantics changed under
  the hood).
- `CORSMiddleware`:
  - `allow_credentials=True` (required for cookies).
  - `allow_origins` must be an explicit list — `*` is incompatible with
    credentialed cookies. `CORS_ORIGINS` becomes mandatory in production; the
    code defaults to a safe local-dev list (`http://localhost:5173`,
    `http://localhost:3000`).

### CLI helper

`backend/app/scripts/hash_password.py` — runnable via
`python -m app.scripts.hash_password`. Uses `getpass` to read the password
without echo, prints the argon2 hash, exits.

### Tests

Add to `backend/tests/`:

- `test_auth_login_success` — correct creds set the cookie, follow-up
  `/api/auth/me` succeeds.
- `test_auth_login_bad_password` — 401, no cookie set.
- `test_auth_login_bad_username` — 401, identical body to bad-password case.
- `test_auth_logout_clears_session` — cookie cleared, subsequent `/me` is 401.
- `test_auth_logout_idempotent` — works without an existing cookie.
- `test_protected_endpoint_requires_session` — `/api/price/candles` is 401
  without cookie, 200 with a valid one.
- `test_health_public` — `/api/health` works with no cookie.
- `test_login_rate_limit` — after 10 failures the 11th attempt returns 429.

## Frontend implementation outline

### `frontend/src/auth.ts` (new)

```ts
export async function login(username: string, password: string): Promise<{username: string}>;
export async function logout(): Promise<void>;
export async function me(): Promise<{username: string} | null>;
```

All three use `credentials: "include"`.

### `frontend/src/api.ts` (rewrite)

- Drop `VITE_API_TOKEN` and `authHeaders()`.
- Introduce `apiFetch(path, init?)` that:
  - Prefixes `API_BASE`.
  - Sets `credentials: "include"`.
  - On `response.status === 401`, dispatches `window.dispatchEvent(new Event("auth:expired"))` before throwing.
- All existing exported fetchers (`fetchCandles`, `fetchExchangeFlows`, …)
  switch to `apiFetch`.

### `frontend/src/components/LoginPage.tsx` (new)

- Centered card on the dashboard background.
- `username` and `password` fields; submit on Enter.
- Inline error message ("Invalid credentials" or "Too many attempts, try
  again in N minutes" on 429).
- Submit button disabled while the request is in flight.
- On success, calls `onSuccess()` — supplied by `AuthGate`.

### `frontend/src/components/AuthGate.tsx` (new)

- On mount, calls `me()`.
- While loading: full-page spinner.
- If authenticated: renders children, exposes `{username}` via a small context
  so `Topbar` can show it.
- If not authenticated: renders `<LoginPage onSuccess={refetch}>`.
- Listens for `auth:expired` and flips back to the unauthenticated state.

### `frontend/src/App.tsx`

Wrap the dashboard in `<AuthGate>`. No other behaviour changes.

### `frontend/src/components/Topbar.tsx`

Add a right-aligned cluster: `Logged in as <username>` · `[Logout]`. The
button calls `logout()` then dispatches `auth:expired`.

  - Add `AUTH_PASSWORD_HASH=` (with a comment pointing at the CLI helper).

- `CLAUDE.md`: remove the bearer-token references, add a one-paragraph "Auth"
  section pointing to this spec and `docs/auth-setup.md`.

## Migration / rollout

1. Add new settings, leave old `api_auth_token` reading temporarily so the
   stack still boots.
2. Land backend auth router + tests.
3. Land frontend `AuthGate` + login page.
4. Remove the `api_auth_token` setting and the static-bearer code path.
5. Update `.env.example` and operator docs.

For deployment, the operator must:

- Run `python -m app.scripts.hash_password`, paste the output into `.env`.
- Restart the api container.
- The first browser hit will land on the login page.

## Risks & mitigations

- **Cookie not sent in dev** — Vite proxy forwards same-origin, so cookies
  work locally. If `VITE_API_URL` is set to a different origin, the operator
  must include that origin in `CORS_ORIGINS` and serve over HTTPS so `Secure`
  cookies are accepted.
- **Lost password** — single account, so recovery means SSH to the host and
  re-running the hash CLI to overwrite `AUTH_PASSWORD_HASH`. Documented.
- **Redis flush wipes sessions** — acceptable; users just log in again.
- **Rate limiter behind a proxy** — must trust `X-Forwarded-For` only when the
  proxy is known. v1 reads `request.client.host`; if the API ever sits behind
  a proxy that hides the real IP, switch to a vetted forwarded-for parser.
